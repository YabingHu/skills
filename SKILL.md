# Python 代码检视

## 检视流程

1. 先确认检视范围：PR diff、提交范围、暂存区变更，或用户指定的文件。优先检视变更代码和直接受影响的调用点，不要一开始就泛扫全仓。
2. 当用户提到公司规范、团队约定、committer 检视、Python 语言检视，或要求包含规范符合性时，读取 `references/company-python-standards.md`。
3. 先看行为正确性，再看风格。优先发现会导致结果错误、崩溃、安全暴露、线上不稳定、数据丢失或缺少测试保障的问题。
4. 根据变更风险检查测试、类型标注、输入校验、日志、异常处理和回滚/清理路径是否足够。
5. 将问题区分为真实缺陷、公司规范违反和可选改进。不要把个人偏好的风格意见包装成正确性问题。

## 检视优先级

按下面顺序检视：

1. 正确性：错误的控制流、边界条件、错误假设、状态污染、意外可变对象、API 契约变化、序列化/反序列化错误。
2. 边界和异常：空输入、`None`、异常数据、重试路径、部分失败、异常后的资源清理、错误信息是否隐藏关键上下文。
3. 类型和接口：类型标注不匹配、不安全转换、动态属性、返回值不清晰、公共接口变化、dataclass/model 字段不一致。
4. 资源管理：文件句柄、网络会话、数据库游标、锁、临时文件、子进程、内存增长、超时处理。
5. 异步和并发：遗漏 `await`、在事件循环中执行阻塞操作、竞态条件、取消处理、共享可变状态、线程/进程安全。
6. 安全和数据处理：注入风险、不安全反序列化、路径穿越、凭证泄露、权限校验、隐私数据日志、依赖可信度。
7. 性能：意外的平方复杂度、重复 I/O、低效数据库查询、不必要的全量物化、缓存失效、高基数日志。
8. 测试和可观测性：缺少回归测试、断言过弱、依赖时间导致不稳定、失败路径不可观测、日志/指标不足以定位问题。
9. 可维护性：函数职责过宽、重复逻辑分叉、隐藏副作用、模块边界不清晰、mock 过于脆弱。
10. 公司规范：按 `company-python-standards.md` 执行；只有规则明确或风险具体时才作为阻塞问题提出。

## 证据要求

- 每条发现尽量包含文件和行号。
- 说明失败模式，而不只是描述“代码味道”。
- 对行为缺陷，优先给出具体输入、触发条件或影响路径。
- 可行时运行仓库已有测试或静态检查；如果没有运行，必须明确说明。
- 格式化问题应优先交给 formatter/linter，不要手工刷低价值评论。
- 避免主观命名、猜测式重构、与本次变更无关的大范围架构评论。

## 输出格式

先列问题，按严重程度排序。建议使用下面格式：

```text
发现
- [P1] path/to/file.py:123 - 简短标题
  问题：具体哪里不对。
  影响：什么条件下会失败，失败后果是什么。
  建议：可执行的修复建议。
  分类：正确性 | 公司规范 | 测试 | 安全 | 性能 | 可维护性
```

严重程度：

- `P0`：会导致线上故障、数据丢失、安全事故，或确定发生的严重失败。
- `P1`：高概率缺陷、接口契约破坏、关键校验缺失，或重要公司规范违反。
- `P2`：中等正确性风险、缺少关键测试、近期会造成维护成本的问题。
- `P3`：轻量建议；只有明确有价值且可执行时才提出。

如果没有发现可行动问题，直接说明“未发现可行动问题”，并列出残余风险，例如未运行测试、只检视了 diff、未覆盖某些调用链。

## 公司规范使用方式

当公司规范适用时：

1. 读取 `references/company-python-standards.md`。
2. 在发现中引用对应规则名或章节。
3. 区分“强制规则”和“建议规则”。
4. 如果规范文件仍是占位模板，不要编造公司规则；可以把问题作为通用 Python 检视建议提出，或向用户确认规则内容。

## 常用本地命令

优先使用仓库已有工具链。选择命令前，先检查 `pyproject.toml`、`setup.cfg`、`tox.ini`、`noxfile.py`、`.pre-commit-config.yaml`、`requirements*.txt`、`Makefile` 等配置。

仓库支持时可使用：

```bash
python -m pytest
python -m pytest path/to/test_file.py
python -m ruff check .
python -m mypy .
python -m pyright
python -m pre_commit run --all-files
```

不要为了检视临时安装新工具，除非用户明确要求，或仓库文档已经说明这是标准流程。
