# 公司 Python 语言规范

这个文件是公司 Python 检视规则的可维护入口。请把占位内容替换为你们团队真实执行的规范。

## 规则填写格式

建议每条规则使用下面格式：

```text
规则：简短规则名
级别：强制 | 建议
适用范围：服务代码 | 公共库 | 脚本 | 测试 | 全部 Python 代码
要求：检视时需要执行的具体要求。
评论方式：违反时应该如何写 review comment。
示例：可选，给出正例/反例。
```

## 代码风格和格式化

- 规则：格式化工具优先
  级别：建议
  适用范围：全部 Python 代码
  要求：空白、导入顺序、简单格式化问题优先交给 formatter/linter，不建议手工逐条评论。
  评论方式：只有当未运行规定工具、会导致 CI 失败，或格式问题影响可读性时再评论。

## 类型标注

- 规则：公共接口类型标注
  级别：占位
  适用范围：公共函数、服务边界、库 API
  要求：在这里补充公司对参数类型、返回值类型、Optional、泛型、类型别名的具体要求。
  评论方式：占位未替换前，不要作为公司强制规范评论。

## 异常处理

- 规则：禁止静默吞掉宽泛异常
  级别：建议
  适用范围：全部 Python 代码
  要求：避免无条件 `except Exception` 后不记录、不重新抛出、也没有明确降级逻辑。
  评论方式：当失败会被调用方、监控或运维完全感知不到时，需要评论。

## 日志规范

- 规则：禁止日志泄露敏感信息
  级别：强制
  适用范围：服务代码、脚本、包含真实凭证或真实数据的测试
  要求：不得记录凭证、token、个人敏感信息、未脱敏请求体或响应体。
  评论方式：按安全/数据处理风险评论，不要仅作为风格问题。

## 模块和 API 边界

- 规则：边界兼容性
  级别：占位
  适用范围：服务 API、公共库、模型 schema、跨模块调用
  要求：在这里补充公司对兼容性、schema 迁移、模块归属、跨团队接口变更的规定。
  评论方式：只有规则明确或 diff 确实破坏现有调用方时，才作为阻塞问题。

## 测试要求

- 规则：行为变更需要测试
  级别：建议
  适用范围：生产代码变更
  要求：行为发生变化时，应补充回归测试；如果不补测试，需要有明确理由。
  评论方式：要求围绕变更行为和边界条件补充有针对性的测试，不要泛泛要求提高无关覆盖率。

## 禁止或慎用模式

在这里补充公司明确禁止或需要升级确认的内容，例如：

- 禁用 API 或库。
- 依赖引入规则。
- 不允许的序列化/反序列化方式。
- 数据库访问约束。
- 权限、认证、审计相关要求。
- 特定框架的分层或目录约定。
